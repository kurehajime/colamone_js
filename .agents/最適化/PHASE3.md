# フェイズ3: 既存JavaScript思考ロジックの高速化手法

## 対象

- `src/static/game/Ai.ts`
  - `Aijs.thinkAI`
  - `Aijs.deepThinkAllAB`
- `src/static/game/Rule.ts`
  - 合法手生成、終局判定、手詰まり判定
- `src/static/game/Eval.ts`
  - 盤面評価
- `src/model/GameState.ts`
  - AI呼び出し時の探索深度決定

## 思考ロジックの概要

`GameState.ai()` は現在の難易度 `level` と終盤補正 `plus` から探索深度を決め、`Aijs.thinkAI()` を呼び出す。
`ENABLE_WASM` は `false` なので、通常はJavaScript実装だけが使われる。

`Aijs.thinkAI()` は盤面を `Int8Array` にコピーし、評価パラメータを補完したうえで `Aijs.deepThinkAllAB()` に処理を渡す。
`deepThinkAllAB()` は現在手番の合法手を全生成し、再帰的に相手番の手を探索して最善手と評価値を返す。

## 既存の高速化手法

### JavaScript実装レベルの細かい高速化

このコードには、探索アルゴリズム自体とは別に、JavaScriptの実行時コストを下げるための書き方がいくつか入っている。
特に `Rule.ts` と `Eval.ts` は、読みやすさよりもホットパスの単純さを優先している箇所がある。

#### `Int8Array` による盤面表現

盤面は `MapArray = Int8Array` として扱われる。
駒の値は `-8..8` に収まるため、signed 8-bit の範囲で十分に表現できる。

利点:

- 通常の `number[]` やオブジェクトより、要素型が固定される。
- 盤面コピーを `new Int8Array(wkMap)` で短く書ける。
- 盤面値が整数であることを表現上も強制できる。
- wasm呼び出し時にも `Int8Array` のまま渡しやすい。

注意点:

- `Int8Array` は範囲外の値を入れると8bit整数に丸められるため、駒番号を拡張する場合は前提を見直す必要がある。
- 盤面インデックスは `0..55` のように10進表記の座標をそのまま使うので、配列長に対して未使用領域がある。
- 未使用領域を詰めた36要素配列に変えると、座標計算や既存の手生成順が変わりやすい。

#### `~~` による整数化

`Rule.getCanMovePanelX()` と `Rule.hasCanMovePanelX()` では、座標計算に `~~` が使われている。

例:

- `~~(panel_num / 10)`
- `~~(panel_num % 10)`
- `~~(i % 3)`
- `~~(i / 3)`

`~~` は数値を32bit整数に変換する書き方で、ここでは `Math.floor()` の代わりとして使われている。
対象値は `panel_num >= 0`、`i >= 0` なので、負数での丸め方向差を気にしなくてよい。

利点:

- `Math.floor()` 呼び出しより短く、関数呼び出し形にならない。
- 値が整数として扱われることをJITに伝えやすい。

注意点:

- `~~` は32bit signed integerへの変換なので、大きい値ではオーバーフローする。
- 負数では `Math.floor()` と結果が異なる場合がある。
- このコードでは盤面座標と9方向ループだけに使っているため安全。

#### `| 0` によるint32化

`Eval.evalMap()` や `Rule` の各処理では、`| 0` で値をint32に寄せている。

例:

- `Rule.NUMBERS[i] | 0`
- `wkMap[panel_num] | 0`
- `wkMap[idx] | 0`
- `return (ev | 0)`

利点:

- typed arrayから読んだ値や評価値を整数として扱う意図が明確になる。
- 小さい整数だけを扱うホットパスでは、浮動小数の混入を避ける書き方になっている。

注意点:

- `| 0` も32bit signed integerへの変換なので、大きな評価値を扱うようになった場合は桁あふれを確認する必要がある。
- 現在の評価値と終局スコアは32bit範囲内なので問題ない。

#### 負数の駒番号を正の配列インデックスへ寄せる

駒の移動テーブル `Rule.PIECES` は、駒番号 `-8..8` を `number + 8` で `0..16` にずらして参照する。

例:

- `Rule.PIECES[number + 8][i]`

これにより、先手と後手の駒を同じ配列テーブルで扱える。
負数キーを持つオブジェクトや `Map` を使わず、通常の配列インデックスだけで移動パターンを引ける。

評価関数でも後手の駒は `evalparam[-1 * p]` で正の駒番号へ戻してから参照している。

例:

- 先手: `evalparam[p][line]`
- 後手: `evalparam[-1 * p][line] * -1`

利点:

- 駒の符号で所属を表し、絶対値で種類を表す設計になっている。
- 配列アクセスが常に非負整数インデックスになる。
- 先手・後手で評価テーブルを二重化しなくてよい。

注意点:

- `0` は空マスなので、`PIECES[8]` は空の移動パターンとして用意されている。
- 駒番号体系を変えると `+ 8` のオフセットも同時に変える必要がある。

#### ループ展開によるゴール列集計

`Rule.isEndX()` と `Rule.isDraw()` では、ゴール列の集計が配列ループではなく個別の `if` で展開されている。
コード内にも「ループだと遅いので展開」とコメントがある。

展開されている対象:

- 先手側ゴール: `0, 10, 20, 30, 40, 50`
- 後手側ゴール: `5, 15, 25, 35, 45, 55`

利点:

- ループ変数、配列参照、コールバックなどの追加コストがない。
- JITが分岐と加算を単純な直列処理として最適化しやすい。
- `isEndX()` と `isDraw()` は探索中に非常に多く呼ばれるため、小さい差でも効きやすい。

注意点:

- 盤面サイズやゴール列が変わると修正漏れが起きやすい。
- 読みやすさは落ちるため、ゲームルールが固定である前提の最適化。

#### 固定長ループと固定テーブル

合法手生成は、駒ごとの処理を分岐で分けず、`PIECES` の3x3テーブルを9回だけ見る。

例:

- `for (let i = 0; i < 9; i++)`
- `if (Rule.PIECES[number + 8][i] === 0) continue`

盤面走査も `Rule.NUMBERS` の36要素だけを `0..35` の固定範囲で見る。

例:

- `for (let i = 0; i <= 35; i++)`
- `const panel_num = Rule.NUMBERS[i] | 0`

利点:

- ループ回数が固定で、分岐構造も安定している。
- `for...of` や高階関数を使わず、単純なカウンタループになっている。
- 実際に存在する36マスだけを走査し、`Int8Array` の未使用添字を無視できる。

注意点:

- `Rule.NUMBERS` の順序は合法手生成順にも影響する。
- 順序を変えるとアルファベータの枝刈り量や同評価時の選択手が変わる。

#### 配列生成を避ける合法手有無判定

手詰まり判定では、合法手配列を作る `getCanMovePanelX()` ではなく、真偽値だけを返す `hasCanMovePanelX()` を使っている。

`hasCanMovePanelX()` は動けるマスを1つ見つけた時点で `true` を返す。
これにより、手詰まり判定で毎回 `number[]` を作るコストを避けている。

利点:

- 配列確保を避けられる。
- 合法手を全部調べる前に終われる。
- `isNoneNode()` のような頻出判定に向いている。

注意点:

- `getCanMovePanelX()` と `hasCanMovePanelX()` は似たロジックを二重に持っている。
- 最適化時に片方だけ変更すると、合法手判定の整合性が崩れる可能性がある。

#### オブジェクトではなくタプルと配列を返す

探索結果や手は、名前付きオブジェクトではなく配列タプルで表現されている。

例:

- `Hand = [number, number]`
- `HandNode = [Hand, MapArray]`
- `deepThinkAllAB()` の返り値: `[Hand | undefined, number]`

利点:

- オブジェクトプロパティ名を持たず、構造が小さい。
- 探索中に大量生成される値の形が単純。

注意点:

- `hand[0]`, `hand[1]` の意味を型名や周辺文脈で読む必要がある。
- 可読性は名前付きプロパティより低い。

#### ビット演算の優先順位に依存する短い書き方

`GameState.ai()` ではログ出力に `end - start | 0` が使われている。
これは計測値を整数ミリ秒に丸めるための書き方。

例:

- `` `time:${end - start | 0}` ``

注意点:

- 表示用の丸めであり、探索自体の高速化ではない。
- 可読性は `Math.trunc(end - start)` より低い。

### アルファベータ枝刈り

`Aijs.deepThinkAllAB()` は `a` と `b` を持ち回り、探索中の最善値が境界を超えた時点でループを `break` する。
これは既存ロジックで最も大きい探索量削減になっている。

注意点:

- 手の順序は `Rule.getNodeMap()` の生成順に依存する。
- 手の順序を変えると探索回数や実行時間だけでなく、同評価時に選ばれる手も変わる可能性がある。

### 終局の早期判定

各候補手を指した直後に `Rule.isEndX()` を呼び、必勝なら即座にその手を返す。
必敗なら、その候補を深く読まずに `continue` する。

これにより、明らかな勝敗局面で残り深度まで再帰しない。

### 手詰まり局面の打ち切り

`Rule.isNoneNode()` が真の場合、評価を `0` として再帰を止める。
合法手がない局面では子ノード生成を避けられる。

### 評価前の終局判定

`Eval.evalMap()` は通常評価に入る前に、次を先に判定する。

- 引き分け
- 終局

これにより、終局局面では36マス分の評価ループを回さずに固定スコアを返す。

### 終盤だけ探索深度を補正

`GameState.ai()` は `Rule.getNodeCount(this.map) / 2` を使って局面の手数規模を見積もり、難易度ごとに `plus` を加算または減算する。

例:

- Lv.5/Lv.6 では序盤に `plus--` して深く読みすぎない。
- 終盤は `plus++` して読みを深くする。

これは探索時間を局面の複雑さに合わせるための実行時間制御になっている。

### 負け確定時の浅い読み直し

`Aijs.thinkAI()` は深い探索結果が現在手番にとって `-999999` の負け評価だった場合、depth 1で読み直す。

これにより、どの手も負ける局面では深い探索で得た負け筋より、直近の局面評価に基づく手を選び直す。
速度最適化というより、負け局面での手選択を調整する制御に近い。

## 次フェイズ以降の注意点

フェイズ2の計測は実行時間だけを見る想定なので、次の最適化では次を守ると比較しやすい。

- 探索深度は変更しない。
- 合法手の生成順は変更しない。
- アルファベータの枝刈り条件は変更しない。
- 評価値が同じ時の手選択順は変更しない。
- `thinkAI` の返す手とスコアは既存テストで確認する。

探索回数を変えない方針なら、候補になりやすいのは以下。

- 再帰中の一時配列やオブジェクト生成を減らす。
- `Rule.isEndX()` / `Rule.isDraw()` / `Rule.isNoneNode()` の重複走査を減らす。
- `getNodeMap()` が全子局面を先に配列化する構造を、手順を保ったまま軽量化する。
- `JSON.stringify()` を使う千日手処理はAI探索内では使われていないため、思考時間最適化の主対象ではない。
