# フェイズ5: JavaScript実装レベルの追加最適化

## 方針

フェイズ5では、探索深度、合法手の生成順、アルファベータ枝刈り条件、評価ロジックを変えない。
対象はJavaScript実装レベルの細かい実行コストだけに限定する。

## 変更内容

### 9方向の座標差分を事前計算する

`Rule.getCanMovePanelX()`、`Rule.hasCanMovePanelX()`、`Rule.getNodeMap()` では、これまで9方向ループの中で毎回次の計算をしていた。

- `~~(i % 3) - 1`
- `~~(i / 3) - 1`
- `target_x * 10 + target_y`

これを `Rule.MOVE_X`、`Rule.MOVE_Y`、`Rule.MOVE_OFFSET` の `Int8Array` に事前計算した。

```ts
static readonly MOVE_X = new Int8Array([-1, 0, 1, -1, 0, 1, -1, 0, 1])
static readonly MOVE_Y = new Int8Array([-1, -1, -1, 0, 0, 0, 1, 1, 1])
static readonly MOVE_OFFSET = new Int8Array([-11, -1, 9, -10, 0, 10, -9, 1, 11])
```

これにより、合法手生成のホットパスで剰余、除算、整数化、移動先index計算を繰り返さず、固定テーブル参照だけで座標差分と移動先indexを得る。

### 駒移動テーブルを平坦化する

既存の `Rule.PIECES[number + 8][moveIndex]` は、駒番号の配列を引いてから方向の配列を引く二段アクセスだった。
追加で `Rule.PIECE_MOVES` を作り、`(number + 8) * 9 + moveIndex` の一次元indexで参照する。

```ts
static readonly PIECE_MOVES = new Int8Array(Rule.PIECES.flat())
```

`Rule.PIECES` は既存の意味を残すため維持し、ホットパスでは `PIECE_MOVES` を使う。

### 同色判定の掛け算を避ける

合法手判定では、これまで `target_number * number > 0` で自コマ判定をしていた。
変更後は `target_number > 0` / `target_number < 0` で分岐し、現在の駒の符号とゴール列条件を直接見る。

これにより、9方向ループ内の掛け算を避ける。
条件の意味は変えていない。

### `Rule.NUMBERS` 参照をローカル化する

`isEndX()`、`isNoneNode()`、`getNodeMap()`、`Eval.evalMap()` の36マス走査で、`Rule.NUMBERS` をローカル変数に受けるようにした。
クラス静的プロパティ参照をホットループ内で繰り返さないための小さい最適化。

### `getNodeMap()` の中間配列を減らす

`Rule.getNodeMap()` は、各駒ごとに `Rule.getCanMovePanelX()` を呼び出して `number[]` の合法手配列を作ってから、子局面を作っていた。

変更後は、`getNodeMap()` 内で同じ順序の9方向ループを直接回し、合法手が見つかった時点で子局面を生成する。
これにより、探索中に大量に作られる一時的な `canMove` 配列を避ける。

維持していること:

- 盤面走査順は `Rule.NUMBERS[0..35]` のまま。
- 方向走査順は `moveIndex = 0..8` のまま。
- 移動可否条件は `getCanMovePanelX()` と同じ。
- 子局面の生成順は変更しない。

### ループ内の静的参照をローカル化する

9方向ループでは、次の静的参照をローカル変数に持つようにした。

- `Rule.MOVE_X`
- `Rule.MOVE_Y`
- `Rule.MOVE_OFFSET`
- `Rule.PIECE_MOVES`

クラス静的プロパティの参照をホットループ内で繰り返さず、単純なローカル変数参照に寄せる。

## 検討したが見送ったもの

### 終局判定と手詰まり判定の統合

`Aijs.deepThinkAllAB()` では `Rule.isEndX()` の後に `Rule.isNoneNode()` を呼ぶ経路があり、内部的に重複判定を減らす余地がある。
ただし、これは関数呼び出し構造と判定のまとまりを変える。

今回は「ロジックが変わるような変更は加えない」という方針に合わせ、探索の見え方や判定境界に影響しにくい言語レベルの微修正だけを採用した。
